# Improve Display UX

## Status
done

## Priority
medium

## Description
The terminal output produced by `pi setup` and `pi run` is functional but lacks polish. Steps that are less interesting (already-installed installers, silent bash steps) clutter the output equally with important information. This project improves readability through better colour use, explicit step execution logging, and control over verbosity via per-step flags and CLI flags.

A separate but related concern: after `pi setup` modifies shell config (e.g. installs shortcuts into `.zshrc`), the changes are not reflected in the running shell. This project also adds the ability for setup to source the shell config in the parent process, and more broadly allows any step to be executed in the parent shell — which is the foundation for features like activating Python virtualenvs from a shortcut.

## Goals
- Terminal output uses colour intentionally: dim/grey for already-satisfied steps, bold/green for work actually done, red for failures
- Every step that executes (including `bash:` steps inside `run:` automations) produces a visible trace line by default
- Steps can opt out of printing with `silent: true`; the `--loud` flag overrides all silencing
- After `pi setup`, the parent shell automatically re-sources the relevant rc file (`.zshrc` / `.bashrc`) so shortcuts are immediately available without a new terminal
- Steps can declare `shell: parent` to run in the invoking shell (e.g. `cd`, `source`, `activate`)

## Background & Context
Reported pain points from real usage of `pi setup` on the bat project:
1. The output shows installer status lines but is silent about subsequent `bash:` steps — it is not obvious that `rustup component add rustfmt clippy` ran at all.
2. All lines have equal visual weight; already-installed tools and newly-installed tools look the same.
3. After `pi setup` installs shell shortcuts, the user must open a new terminal or manually `source ~/.zshrc` — this breaks the "zero friction" promise.
4. There is no mechanism to run a step in the parent shell, which blocks use-cases like virtualenv activation.

## Scope

### In scope
- Colour-coded output for setup/run (dim=skipped/already-done, normal=running, green=success, red=fail)
- Per-step `silent: true` flag that suppresses its trace line
- `--loud` CLI flag that forces all steps (including silent ones) to print
- Auto-source of rc file in parent shell after `pi setup` completes
- Per-step `shell: parent` flag for steps that must run in the parent process
- Tests (unit + docker integration) for all of the above

### Out of scope
- Structured / machine-readable output formats
- TUI / interactive progress spinners
- Changes to `pi run` verbosity beyond step trace lines

## Success Criteria
- Running `pi setup` on a project with an installer automation shows dimmed lines for already-installed tools and a normal/green line for any tool actually installed
- A `bash:` step inside a `run:` automation prints a trace line `  → bash: <first 60 chars of command>` before executing, unless marked `silent: true`
- `pi setup --loud` prints trace lines even for steps marked `silent: true`
- After `pi setup` finishes (and shell shortcuts are installed), the calling shell has the shortcuts available immediately without manual sourcing
- A step with `shell: parent` executes in the parent shell process; a `cd` step changes the parent's working directory
- All new behaviour is covered by tests; `go test ./...` passes

## Notes
- Parent-shell execution requires research: the canonical technique is printing commands to a tmp file and having the shell wrapper `eval` the file. The shell shortcut wrappers generated by `pi shell` are the right place to wire this up.
- The `shell: parent` feature is important for Python virtualenv workflows — keep the design general enough to support `source venv/bin/activate`.
- Colour output should respect `NO_COLOR` env var and degrade gracefully when stdout is not a TTY.
