# Bug: `parent_shell: true` with `cd` Has No Effect

## Type
bug

## Status
done

## Priority
high

## Project
standalone

## Description
A `bash` step with `parent_shell: true` running a `cd` command does not change the working directory of the calling shell. The shell stays in its original directory regardless of how PI is invoked.

**Reproduction 1 — via `pi run` directly:**

```yaml
# .pi/project/move-to-dir.yaml
name: project/move-to-dir
description: Moves the shell into the project directory
steps:
  - bash: cd ~
    parent_shell: true
```

```bash
$ pwd
/Users/yotam/projects/pi-test

$ pi run project/move-to-dir
  → bash: cd ~

$ pwd
/Users/yotam/projects/pi-test   # ← unchanged, expected ~
```

**Reproduction 2 — via shell shortcut (also broken):**

```yaml
# pi.yaml
project: pi-test
shortcuts:
  pitestcd: project/move-to-dir
```

```bash
$ pwd
/Users/yotam/projects/pi-test

$ pitestcd
  → bash: cd ~

$ pwd
/Users/yotam/projects/pi-test   # ← still unchanged — shortcut also broken
```

**Root cause analysis:**

There are two distinct problems:

**Problem A — `pi run` called directly:** `PI_PARENT_EVAL_FILE` is not set, so the PI binary silently drops the step without any warning. The user gets no feedback that their `parent_shell: true` step did nothing.

**Problem B — shell shortcut also broken:** Even with `PI_PARENT_EVAL_FILE` set by the shortcut wrapper, the `cd` still has no effect. The shortcut wrapper sources the eval file after PI exits, which should work. This is a deeper bug — either the command is not being written to the eval file, or the source is happening in a subshell context that doesn't affect the calling shell. Needs debugging. (Note: `source venv/bin/activate` is the only tested `parent_shell` use case — it's possible `cd` specifically isn't being written to the eval file, or that the shortcut wrapper has a structural issue.)

**The proper long-term fix — global `pi()` wrapper:**

`pi shell` should install a global `pi()` shell function alongside project shortcuts. This wraps every `pi` invocation with the eval pattern so `parent_shell: true` works for all `pi run` calls, not just shortcuts:

```bash
# Generated by: pi shell (global wrapper)
function pi() {
  local _pi_eval_file
  _pi_eval_file="$(mktemp)"
  PI_PARENT_EVAL_FILE="$_pi_eval_file" command pi "$@"
  local _pi_exit=$?
  if [ -s "$_pi_eval_file" ]; then
    source "$_pi_eval_file"
  fi
  rm -f "$_pi_eval_file"
  return $_pi_exit
}
```

`command pi` calls the real binary to avoid infinite recursion. With this in place, `pi run project/move-to-dir` works identically to a shortcut.

## Implementation Notes

### Root cause of Problem B

The generated shortcut function had:
```bash
PI_PARENT_EVAL_FILE="$_pi_eval_file" (cd "/path" && pi run ...)
```

This is a **shell syntax error** in both bash and zsh. `VAR=value (compound-command)` is not valid — the `VAR=value` prefix only works before a **simple command** (an executable). The shell either rejected it silently or ignored the env var, meaning `PI_PARENT_EVAL_FILE` was never set inside the subshell where `pi` runs.

### Fix

Moved `PI_PARENT_EVAL_FILE` **inside** the subshell as a simple command prefix:
```bash
(cd "/path" && PI_PARENT_EVAL_FILE="$_pi_eval_file" pi run ...)
```

This matches the pattern already used by the `generateSetupHelper` function, which was working correctly.

Both `generateFunction` and `generateFunctionWithInputs` were updated. The `anywhere` variant (no subshell) already had correct syntax since `VAR=value command` is valid.

### Warning behavior change

Previously, `execParentShell()` returned a hard error when `PI_PARENT_EVAL_FILE` was not set, causing the entire automation to fail. Changed to a non-fatal warning printed to stderr, allowing the rest of the automation to continue. The warning includes guidance to run `pi shell`.

### Global `pi()` wrapper

Added `GenerateGlobalWrapper()` which generates a `pi()` shell function using `command pi` to call the real binary (avoids infinite recursion). Written to `~/.pi/shell/_pi-wrapper.sh` (underscore prefix ensures it sorts before project files). The wrapper file is:
- Created/updated during `Install()`
- Removed during `Uninstall()` when the last project is removed
- Excluded from `ListInstalled()` results
