# Parent shell sourcing after setup and parent-shell step execution

## Type
research

## Status
done

## Priority
high

## Project
11-improve-display-ux

## Description
Two related problems require running commands in the **parent shell** (the shell that invoked `pi`):

**Problem 1 — setup doesn't take effect immediately.**
After `pi setup` installs shell shortcuts into `.zshrc`, the user must open a new terminal or manually run `source ~/.zshrc`. This breaks the zero-friction promise: the developer just ran setup and their new shortcuts don't work yet.

**Problem 2 — no way to affect parent shell state.**
Some automations must modify the parent shell's environment — the canonical example is activating a Python virtualenv (`source venv/bin/activate`) or changing directory. Because `pi` is a subprocess, it cannot directly affect the parent shell. There is currently no mechanism for a step to do this.

### Approach to research

The standard Unix technique is:
1. The `pi` binary writes commands to a temp file (e.g. `~/.pi/tmp/parent-eval-<pid>.sh`)
2. The shell wrapper function (generated by `pi shell`) `eval`s that file after `pi` exits
3. `pi` exits with a special exit code OR writes the file only when there is something to eval

The shell shortcut wrappers already generated by `pi shell` are the right place to wire this up. The function currently looks like:
```bash
function bat-install() {
  (cd /path/to/bat && pi run build/install "$@")
}
```
It would become something like:
```bash
function bat-install() {
  local _pi_eval_file
  _pi_eval_file="$(mktemp)"
  PI_PARENT_EVAL_FILE="$_pi_eval_file" pi run build/install "$@"
  if [ -s "$_pi_eval_file" ]; then
    eval "$(cat "$_pi_eval_file")"
  fi
  rm -f "$_pi_eval_file"
}
```

### Part A — Auto-source after `pi setup`

After `pi setup` completes and shell shortcuts are (re)installed:
- Detect the user's shell (zsh / bash) and determine the rc file (`.zshrc` / `.bashrc`)
- Write `source <rc-file>` (or the equivalent) to `PI_PARENT_EVAL_FILE` if set
- Update the `pi setup` wrapper (in the generated `pi-setup` shell helper, or in the main `pi setup` command body) to set `PI_PARENT_EVAL_FILE`
- Note: `pi setup` itself may not be invoked via a shortcut wrapper initially; document the one-time bootstrapping story (first run won't auto-source, subsequent runs will once the shell helper exists)

### Part B — `shell: parent` step flag

Add a `shell: parent` (or `in_parent_shell: true`) field to the step schema. When a step sets this flag:
- `pi` does **not** execute the step itself
- Instead it appends the step's command to `PI_PARENT_EVAL_FILE`
- After `pi` exits, the shell wrapper evals the file, running the command in the parent shell

### Testing

This feature must be tested end-to-end. The Docker test matrix is the right place:
- Add a test scenario that runs `pi setup` inside a container, then verifies that the shell has the shortcuts available in the same shell session
- Add a test for a step with `shell: parent` that does `cd /tmp` and verify the parent shell's cwd changes

## Implementation Notes

### Approach chosen
Used the eval-file pattern described in the task's research section. The implementation has three parts:

**1. Shell codegen (`internal/shell/shell.go`)**
- All shortcut functions now use the eval wrapper: create temp file → set `PI_PARENT_EVAL_FILE` → run `pi` → source file if non-empty → cleanup → preserve exit code
- Added `pi-setup-<project>` helper function that wraps `pi setup` the same way
- Added `PrimaryRCFile()` export for determining which rc file to source

**2. Step schema + executor (`internal/automation/automation.go`, `internal/executor/executor.go`)**
- Added `parent_shell: true` field on steps (YAML key: `parent_shell`)
- Only valid on bash steps, cannot combine with `pipe_to` (validated at parse time)
- `Executor.ParentEvalFile` populated from `PI_PARENT_EVAL_FILE` env var in `cli/run.go` and `cli/setup.go`
- `execParentShell()` writes the command to the eval file instead of executing it
- `AppendToParentEval()` is a public helper for appending commands to the file

**3. Auto-source after setup (`internal/cli/setup.go`)**
- After `pi setup` installs shortcuts, if `PI_PARENT_EVAL_FILE` is set, writes `source <rc>` to the eval file
- Uses `shell.PrimaryRCFile()` to find the user's rc file

### Design decisions
- **Field name**: Chose `parent_shell: true` (not `shell: parent`) because it's a simple boolean flag on a step, not a selector. This is cleaner in YAML and easier to validate.
- **Bash-only**: Only bash steps can be parent_shell — the eval file runs in bash/zsh context, so python/typescript don't make sense. `run:` steps should contain parent_shell steps inside the called automation instead.
- **Error on missing eval file**: If a parent_shell step runs without `PI_PARENT_EVAL_FILE` set, it errors rather than silently executing as a subprocess. This prevents subtle bugs where the user expects parent-shell behavior but doesn't get it.
- **Exit code preservation**: The shell wrapper captures `$?` immediately after `pi` exits and returns it after sourcing the eval file.

### First-run bootstrapping
On the very first `pi setup` (before any shell wrapper exists), the eval pattern isn't active because the user invokes `pi setup` directly, not through a wrapper. The user must `source ~/.zshrc` manually once. After that, `pi-setup-<project>` is available and handles auto-sourcing.

### Edge cases
- Skipped parent_shell steps (via `if:`) don't write to the eval file
- Multiple parent_shell steps append to the same file
- Mixed normal + parent_shell steps work — normal steps execute, parent_shell steps accumulate
- The eval file is cleaned up even if `pi` exits with an error
